# Contributing

Thank you for contributing! This guide explains how to propose changes and collaborate effectively.

## Getting started

1. Fork the repository.
2. Create a branch from `main`:
   - `feature/<short-description>` for new features
   - `fix/<short-description>` for bug fixes
3. Make your changes in small, focused commits.

## Development setup

1. Clone your fork and enter the project directory.
2. Install dependencies using the project's package manager or build tool.
3. Run the test suite and ensure it passes before opening a pull request.
4. If available, run linting/formatting tools and fix all reported issues.

## Coding standards

- Follow the existing code style and naming conventions in the repository.
- Keep functions and classes focused and readable.
- Add or update tests for behavior changes.
- Update documentation when introducing user-facing or developer-facing changes.

## Commit guidelines

- Write clear, descriptive commit messages.
- Prefer atomic commits that each represent one logical change.
- Avoid mixing refactors with functional changes unless necessary.

## Pull request process

When opening a pull request, please include:

- A concise summary of what changed and why.
- Links to related issues (for example, `Fixes #123`).
- Notes about testing performed.
- Screenshots or logs when relevant.

Before requesting review, verify:

- [ ] Tests pass locally.
- [ ] Lint/format checks pass.
- [ ] Documentation is updated as needed.
- [ ] The change is scoped and ready for review.

## Code review expectations

- Be respectful and constructive in review comments.
- Respond to feedback promptly.
- If you disagree with feedback, discuss with rationale and examples.
- Reviewers should focus on correctness, maintainability, security, and clarity.

## Reporting issues

- Use the issue tracker for bugs and feature requests.
- Include reproduction steps, expected behavior, and actual behavior for bugs.
- For security-sensitive issues, use the project's private/security reporting channel if available.

## Questions

If anything is unclear, open an issue and ask for guidance before starting large changes.
